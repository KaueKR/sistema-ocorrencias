--------------------------------------------------------------------------------
LEGENDA DE CARDINALIDADE
--------------------------------------------------------------------------------
  1:1  → um para um         1:N  → um para muitos
  N:1  → muitos para um     N:N  → muitos para muitos (via tabela de junção)

--------------------------------------------------------------------------------
ENTIDADES E ATRIBUTOS  (conforme schema_banco_de_dados.png)
--------------------------------------------------------------------------------

perfis (1:1 com auth.users)
  id              uuid         PK, FK → auth.users(id)
  nome_completo   text
  email           text
  telefone        text
  tipo_usuario    text         (campo legado, migrado para o RBAC)
  avatar_url      text
  ativo           boolean
  criado_em       timestamptz
  atualizado_em   timestamptz
  role_principal  varchar      (cache da role de maior autoridade)

setores
  id                uuid         PK
  nome              text
  descricao         text
  email_responsavel text
  ativo             boolean
  criado_em         timestamptz

categorias
  id              uuid         PK
  setor_id        uuid         FK → setores(id)
  nome            text
  descricao       text
  icone           text         (nome do ícone Ionicons)
  ativo           boolean
  criado_em       timestamptz
  restrito        boolean      (true = visível só p/ roles privilegiadas)

ocorrencias
  id                  uuid         PK
  usuario_id          uuid         FK → perfis(id)   (autor)
  categoria_id        uuid         FK → categorias(id)
  setor_id            uuid         FK → setores(id)
  titulo              text
  descricao           text
  local               text
  urgencia            text         (baixa | media | alta)
  status              text         (aberta | em_analise | em_andamento |
                                     resolvida | cancelada | encerrada)
  responsavel_id      uuid         FK → perfis(id)   (responsável designado)
  observacao_interna  text
  criado_em           timestamptz
  atualizado_em       timestamptz

fotos_ocorrencias
  id              uuid         PK
  ocorrencia_id   uuid         FK → ocorrencias(id)
  url             text         (URL pública do Storage)
  storage_path    text         (caminho no bucket fotos-ocorrencias)
  criado_em       timestamptz

historico_status
  id              uuid         PK
  ocorrencia_id   uuid         FK → ocorrencias(id)
  alterado_por    uuid         FK → perfis(id)
  status_anterior text
  status_novo     text
  observacao      text
  criado_em       timestamptz

suporte
  id              uuid         PK
  perfil_id       uuid         FK → perfis(id)
  tipo            text         (tipo de problema)
  descricao       text
  status          text
  criado_em       timestamptz

--------------------------------------------------------------------------------
RBAC (Controle de Acesso Baseado em Papéis)
--------------------------------------------------------------------------------

roles
  id, nome (único), label, nivel (int4), descricao, ativo, criado_em
  Hierarquia: 1 = maior autoridade.
    1 super_admin | 2 admin_institucional | 3 gestor_setor
    4 tecnico     | 5 professor           | 6 aluno (padrão)

permissoes
  id, codigo (único, padrão "modulo.acao"), descricao, modulo, criado_em
  Módulos: ocorrencias, usuarios, setores, relatorios, sistema.

role_permissoes (N:N entre roles e permissoes)
  role_id       FK → roles(id)
  permissao_id  FK → permissoes(id)
  PK (role_id, permissao_id)

perfil_roles (N:N entre perfis e roles, com escopo de setor)
  id, perfil_id FK → perfis(id), role_id FK → roles(id),
  setor_id FK → setores(id) (opcional), concedido_por FK → perfis(id), criado_em

--------------------------------------------------------------------------------
CICLO DE VIDA DA OCORRÊNCIA (campo status)
--------------------------------------------------------------------------------

  aberta ──► em_analise ──► em_andamento ──► resolvida ──► encerrada
     │                                            ▲
     └──────────────► cancelada ──────────────────┘

Cada transição é registrada na tabela historico_status (auditoria).

--------------------------------------------------------------------------------
ARMAZENAMENTO (Supabase Storage)
--------------------------------------------------------------------------------
  Bucket: fotos-ocorrencias (público)
  As fotos são enviadas em Base64, salvas no bucket e a URL pública é
  registrada em fotos_ocorrencias.url.
